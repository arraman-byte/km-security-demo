Authorization Code (no PKCE)

The classic OAuth2 server-side flow for confidential clients (apps that can keep a secret on a backend). The browser only ever sees an opaque authorization code; the secret-bearing backend swaps it for tokens at the /token endpoint.

When to use: traditional server-rendered web apps with a backend that can safely store client_secret. For SPAs, mobile, CLIs — use Authorization Code + PKCE instead.

Sequence

  1. Browser redirected to /authorize?response_type=code&client_id=...&redirect_uri=...&state=....
  2. User logs in at Keycloak; redirected back with ?code=...&state=....
  3. Backend POSTs to /token with grant_type=authorization_code, code, client_id, and client_secret.
  4. Keycloak returns access_token, id_token, refresh_token.
